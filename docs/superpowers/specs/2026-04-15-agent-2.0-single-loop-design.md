# Agent 2.0（单 Agent 闭环 + 原生 Tool-Calling）设计稿

> 范围版本：`2.0.0`
>
> 目标分支：`develop/2.0`

---

## 1. 背景与目标

当前项目已具备：
- 聊天与流式输出
- JD 网页抽取与分析
- BattleCard 持久化

但尚缺以下 Agent 核心能力：
- Planner/Executor 自主循环
- LLM 原生 tool-calling/function-calling 编排
- 评测闭环与自动化测试

本次 2.0 目标是在不破坏 1.0 产品能力的前提下，新增一个可验证、可解释、可扩展的单 Agent 执行链路。

---

## 2. 2.0 范围（已确认）

### 2.0 必做
- 单 Agent Planner/Executor 闭环（有最大步数、终止条件）
- 原生 tool-calling（OpenAI-compatible 协议）
- 3 个核心工具：
  1. `extract_job_page`
  2. `get_profile`
  3. `save_battle_card`
- 测试与评测闭环（`npm test` + `npm run eval:agent`）

### 2.0 不做（留给 2.1+）
- 多 Agent 协作
- 更多业务工具（如 `search_cards`、`append_review_note`）
- 长期记忆/RAG 检索系统

---

## 3. 方案选型与取舍

### 方案 A：Prompt 工作流增强（不引入原生 tool-calling）
- 优点：改动小，上线快
- 缺点：不可控、不可验证，不满足“原生 tool-calling”目标

### 方案 B：单 Agent + 原生 tool-calling（推荐，2.0 采用）
- 优点：复杂度可控；能体现 Agent 核心能力；便于测试与面试讲解
- 缺点：需要新增编排层与测试基建

### 方案 C：直接多 Agent
- 优点：展示上限高
- 缺点：实现与调试复杂，2.0 风险高

结论：采用 **方案 B**。

---

## 4. 总体架构

新增模块放在 `src/main/services/agent/`：

1. `types.ts`
- 定义 `AgentRunRequest`、`AgentRunResult`、`ToolCallRecord`、`AgentTrace`

2. `tools.ts`
- 工具注册与执行器
- 固定白名单：`extract_job_page`、`get_profile`、`save_battle_card`

3. `planner-executor.ts`
- Agent 主循环
- 每轮行为：
  1. 发送上下文给模型
  2. 解析 `tool_calls`
  3. 执行工具并将结果回填对话
  4. 判断终止（模型给出最终答案或达最大步数）

4. `index.ts`
- 对外暴露 `runAgentWorkflow(...)`

已有模块增强：
- `src/main/services/ai/provider.ts`：扩展 tool-calling 接口
- `src/main/services/ai/openai-compatible.ts`：实现 tool schema、tool_calls 解析
- `src/main/ipc/ai.ts`：新增 `ai:agentRun` IPC
- `src/preload/index.ts`、`src/preload/index.d.ts`、`src/renderer/src/utils/ipc.ts`：暴露 Agent API
- `src/renderer/src/components/chat/ChatPanel.tsx`：新增 Agent 模式入口（最小侵入）

## 5. Tool 设计（2.0 固定白名单）

### 5.1 `extract_job_page`
- 入参：`{ webContentId: number }`
- 行为：复用现有 `webview:extract` 提取逻辑
- 出参：`ExtractedContent`

### 5.2 `get_profile`
- 入参：`{}`
- 行为：读取 `profileStore`，并在工具层做字段收敛
- 出参（最小权限）：
  - `resumeTextSnippet`（截断后）
  - `selfIntroTextSnippet`（截断后）
  - `hasResume`（boolean）
- 不直接返回 profile 的完整原文，降低过度暴露风险

### 5.3 `save_battle_card`
- 入参：结构化字段（公司、岗位、分析结果、来源 URL 等）
- 行为：调用 `cardStore.create` 并统一清洗字段
- 出参：`{ cardId: string, companyName: string, positionName: string }`

### 工具安全约束
- 仅允许白名单工具
- 工具入参做严格校验（缺字段/类型错误直接返回可读错误）
- 不支持任意代码执行，不暴露文件系统工具

## 6. Planner/Executor 循环细节

### 输入
- 用户指令（来自 ChatPanel）
- 上下文（至少包含 `webContentId`，用于页面抽取）

### 循环规则
- `maxSteps = 6`（可配置）
- 每轮：
  1. 调用 `provider.chatWithTools(...)`
  2. 若返回 `tool_calls`：按顺序执行工具，记录 trace，继续下一轮
  3. 若返回最终文本：终止并返回
- 触发兜底终止条件：
  - 超过步数上限
  - 连续无效 tool call
  - 工具执行异常达到阈值

### 工具报错“自愈”提示
当工具返回错误（`tool` 消息）时，附加系统约束提示：
- 先分析错误原因
- 判断是重试、改参，还是向用户说明阻塞点
- 禁止对完全相同参数进行无限重复调用

### 上下文压缩策略（成本与性能）
- 对大字段（如 `extract_job_page.content`）做轮次压缩：
  - 第一次保留原始摘要 + 限长正文
  - 从第二次循环开始只保留“工具输出摘要”与关键字段
- 压缩由编排层负责，模型只看到可控长度上下文

### 输出
- `finalAnswer`
- `trace`（每步的模型决策、工具调用、耗时、错误）
- `artifacts`（如生成的 `cardId`）

## 7. OpenAI-Compatible Tool-Calling 兼容

Provider 增加方法（示意）：
- `chatWithTools(messages, tools, options)`

请求体关键字段：
- `tools`
- `tool_choice`（默认 `auto`）

响应解析：
- 兼容 `message.tool_calls[]`
- 兼容纯文本回答（无工具调用）

保留现有：
- `chat`
- `chatStream`

确保 1.0 功能不回归。

---

## 8. IPC 与前端实时反馈（UX 增强）

在 `ai:agentRun` 流程中增加中间状态回传：
- `ai:agentRun:progress`（阶段进度事件）
- `ai:agentRun:done`
- `ai:agentRun:error`

前端实时渲染示例：
- `[正在提取页面...]`
- `[正在读取简历...]`
- `[正在保存战卡...]`

即使最终回答尚未完成，也能持续反馈执行阶段，降低等待焦虑。

## 9. 并发控制与边界

### 单任务锁（2.0）
- 同一窗口同一时刻只允许一个 Agent run
- 当任务运行中再次点击：
  - 前端按钮置灰
  - 后端返回 `AGENT_BUSY`（不并行执行）

这样可避免 `save_battle_card` 并发写入引发竞态。

### 后续扩展（2.1）
- 可增加“中断当前任务并重启”模式
- 可增加多任务队列

---

## 10. UI 与交互（最小改动）

在 ChatPanel 增加 Agent 触发入口（按钮）：
- 普通聊天继续走原 `chatStream`
- Agent 请求走新 `agentRun`

交互反馈：
- 显示中间进度事件（第 8 节）
- 完成后输出：
  - 最终回答
  - 简要执行轨迹（如“提取页面 -> 读取简历 -> 保存卡片”）

## 11. 测试与评测闭环

### 11.1 自动化测试（必做）

引入测试脚本：
- `npm test`
- `npm run test:watch`（可选）

测试层次：
1. `planner-executor` 单测
- 正常 tool-call 链路
- 达到 `maxSteps` 终止
- 工具报错恢复与失败返回
- 上下文压缩是否生效
- 单任务锁是否生效

2. `tools` 单测
- 参数校验
- 返回结构一致性
- `get_profile` 字段收敛正确性

3. provider 解析单测
- tool_calls 响应解析
- 文本响应解析

### 11.2 评测 Harness（必做）

新增目录：`evals/agent-v2/`
- `cases/*.json`：输入与期望行为
- `runner.ts`：批量跑 case，输出通过率与失败明细

命令：
- `npm run eval:agent`

评测核心指标：
- Tool 使用正确率
- 任务完成率（是否得到最终可用答案）
- 平均步骤数（避免无效循环）
- **幻觉率（Hallucination Rate）**：
  - 定义：`save_battle_card` 入参与 `extract_job_page` 事实不一致的比例
  - 重点校验字段：公司名、岗位名、薪资等关键事实

---

## 12. 错误处理与可观测性

- 每次 Agent run 生成 `runId`
- 关键日志统一前缀：`[agent]`
- 错误分层：
  - 模型错误（HTTP/解析）
  - 工具错误（参数/执行）
  - 编排错误（循环控制）
- 对用户返回可读错误，不泄漏敏感信息（如 API Key）

---

## 13. 验证标准（Definition of Done）

满足以下条件即 2.0 核心达成：
1. 能在应用内触发单 Agent 执行
2. Agent 能在单次任务中调用 3 个工具并完成闭环
3. 前端可实时展示 Agent 中间执行状态
4. 存在自动化测试并可通过 `npm test`
5. 存在评测脚本并可通过 `npm run eval:agent`
6. 评测输出包含幻觉率指标
7. 现有 1.0 功能（普通聊天、提取分析、卡片管理）无回归
